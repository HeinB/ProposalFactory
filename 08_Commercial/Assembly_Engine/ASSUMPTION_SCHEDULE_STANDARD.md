# Assumption Schedule Standard — Client-Facing Format Specification

**Version:** 1.0 | **Date:** 2026-06-22 | **Status:** Approved  
**Authority:** This document is the single authoritative format specification for all client-facing Assumption Schedules generated by the Assembly Engine.  
**Companion documents:** `ASSUMPTION_PUBLICATION_RULES.md` | `ASSUMPTION_GROUPING_RULES.md`

---

## 1. Purpose and Scope

### 1.1 What this document defines

This standard defines:

- The purpose and audience of the client-facing Assumption Schedule
- The canonical document structure (8 sections)
- The entry format for each assumption
- The numbering standard
- The traceability model
- The deduplication rules
- The output format for WP17D and beyond

### 1.2 What the Assumption Schedule is

The client-facing Assumption Schedule is a **commercial protection document** included in every APPSolve fixed-price or managed services proposal. It defines the assumptions under which the proposed price, timeline, and scope have been calculated.

Client receipt and non-rejection of the schedule constitutes commercial acknowledgement of the assumptions.

### 1.3 What it is not

| Not | Reason |
|---|---|
| A project plan | Delivery methodology is in the implementation approach section |
| A requirements document | Requirements are elicited during design; assumptions pre-date design |
| A risk register | Risks are listed separately; assumptions define what we rely on being true |
| An internal audit trail | The Assembly Audit Report is the internal document; this is the external output |
| A pack manifest | Pack names, versions, load order, and suppression logs stay internal |

### 1.4 Primary audience

- **Client procurement team** — reads during RFP evaluation
- **Client project sponsor** — signs off on assumptions at contract stage
- **APPSolve bid manager** — produces the schedule via the Assembly Engine
- **APPSolve commercial team** — uses as contract protection reference

## 2. Document Anatomy

### 2.1 Required document elements (in order)

1. **Document header block** (Section 3)
2. **Engagement context summary** (Section 4)
3. **Section A — Engagement and Commercial Assumptions**
4. **Section B — Project and Implementation Assumptions**
5. **Section C — Application Functional Assumptions** (sub-sections per product in scope)
6. **Section D — Infrastructure Assumptions** *(omit if OCI not in scope)*
7. **Section E — Service Management and Support Assumptions** *(omit if AMS not in scope)*
8. **Section F — Security and Compliance Assumptions**
9. **Section G — Client Responsibilities**
10. **Section H — Explicit Exclusions**

### 2.2 Conditional sections

| Section | Include when |
|---|---|
| D — Infrastructure | OCI Pack triggered (EBS on OCI / Fusion on OCI / OIC on OCI / AMS-managed OCI) |
| E — Service Management | AMS Pack triggered (any managed services engagement) |
| C.2 — Oracle HCM | HCM Base Pack triggered |
| C.3 — Oracle Integration | OIC Pack triggered |
| C.4 — Acumatica ERP | Acumatica Base Pack triggered |
| C.5 — BeBanking | BeBanking Base Pack triggered |

Omit an entire section if no assumptions from that section are in the assembled net output.

## 3. Document Header Block

The following header appears at the top of every client-facing schedule:

```
---
[CLIENT NAME] — ASSUMPTION SCHEDULE
Version: [X.0]
Date: [YYYY-MM-DD]
Tender Reference: [Tender ID]
Engagement: [Brief engagement description]
Assembled: [Assembly Engine version]
---
```

Below the header block, include this standard preamble (verbatim, not editable per tender):

> This Assumption Schedule forms part of APPSolve's proposal and defines the assumptions under which the proposed scope, price, and timeline have been calculated. These assumptions represent the basis on which APPSolve has planned its approach and resources.
>
> Where any of these assumptions prove to be incorrect or where the client's requirements deviate from these assumptions, APPSolve reserves the right to raise a change request to address any resultant impact on scope, cost, or timeline.
>
> Client non-objection to this schedule within [10] business days of proposal submission constitutes acknowledgement of these assumptions for commercial purposes.

**Note:** The bracketed `[10]` may be adjusted per engagement per commercial review. The default is 10 business days.

## 5. Assumption Entry Format

### 5.1 Canonical format

Each assumption is presented as a numbered list entry:

```
N. [Full text of assumption.] *(Ref: ID)*
```

Where:
- `N` = Sequential number within the section (1, 2, 3 ... continuing from prior section)
- `[Full text]` = The complete assumption text from the source pack, exactly as approved
- `*(Ref: ID)*` = Internal source ID in italics in parentheses (e.g., `*(Ref: ERP-GEN-001)*`)

### 5.2 Example entries

```
1. The scope of this engagement is the implementation of Oracle Fusion ERP including General Ledger,
   Accounts Payable, Accounts Receivable, Fixed Assets, and Cash Management for a single legal entity.
   *(Ref: ERP-GEN-001)*

2. APPSolve's proposed approach is based on a standard Oracle Cloud implementation methodology with
   three phases: Design, Build, and Deploy. *(Ref: ERP-GEN-002)*
```

### 5.3 Numbering rules

| Rule | Specification |
|---|---|
| **Numbering scope** | Sequential across the entire document (not reset per section) |
| **Starting number** | Always 1 |
| **Increment** | +1 per assumption |
| **Section headers** | Do not consume a number |
| **Sub-section headers** | Do not consume a number |
| **Suppressed assumptions** | Do not appear; do not consume a number |
| **Replacement assumptions** | Appear in their natural section position; consume a number |

### 5.4 Rationale for sequential numbering

Sequential numbering across the document (not per section) is chosen because:

1. **Client referencing** — The client can reference "assumption 47" in correspondence without ambiguity
2. **Contract integration** — Fixed-price contracts that incorporate assumption schedules reference numbered items
3. **Change request linkage** — CRs can cite "deviation from assumption [N]"
4. **Version comparison** — When schedules are revised, the added/removed assumption number can be tracked

Internal source ID (Ref) provides traceability back to the pack without requiring the client to understand internal pack structure.

### 5.5 Reference ID visibility

The `*(Ref: ID)*` suffix is:
- **Included** in all client-facing schedules (provides auditable traceability)
- **Not explained** to the client (internal reference format; clients do not need to understand it)
- **Used internally** to trace any client query back to the exact approved assumption

## 6. Suppressed Assumptions — Client Document Treatment

### 6.1 Rule

**Suppressed assumptions are never shown in the client-facing schedule.**

The suppression mechanism (Rules S1–S4) is an internal Assembly Engine concept. Clients receive only the final, clean set of active assumptions.

### 6.2 Replacement assumptions

When assumption X is suppressed and replaced by assumption Y:
- X does not appear anywhere in the client schedule
- Y appears in its natural section position
- No notation, strikethrough, or reference to the suppression is shown
- The client sees a single coherent assumption, not the replacement history

### 6.3 Where suppression is documented

The suppression register and suppression audit trail are contained in the **Assembly Audit Report** (internal document). The Audit Report accompanies the tender file but is NOT shared with the client.

## 7. Traceability Model

### 7.1 Two-layer traceability

| Layer | Location | Audience |
|---|---|---|
| **Client layer** | Assumption number + Ref ID in client schedule | Client for commercial reference |
| **Internal layer** | Full suppression log + pack + version + rule in Audit Report | Internal bid manager, BU Lead, commercial team |

### 7.2 What the client-facing Ref ID traces

The `Ref: ID` in the client schedule maps to:

| Element | Example | Encoded in Ref ID |
|---|---|---|
| Source pack | Oracle ERP Pack | Prefix `ERP-` |
| Source section | General Ledger | Code `GL` |
| Sequence within section | First assumption in GL | `-001` |

Full example: `ERP-GL-001` → Oracle ERP Pack / General Ledger / Assumption 1

The client does not need to understand the encoding. It serves as a unique opaque identifier for commercial and contractual purposes.

### 7.3 What the Audit Report traces (internal)

The Audit Report records the following per assembled assumption:

| Field | Content |
|---|---|
| Client schedule number | Sequential number in published document |
| Source ID | e.g., ERP-GL-001 |
| Source pack name | Oracle ERP Pack |
| Source pack version | 1.0 |
| Source section | General Ledger |
| Assembly rule applied | Rule A (base first) |
| Suppression rule applied | None / S1 / S2 / S3 / S4 |
| Replacement for | (if a replacement assumption) e.g., replaces AMS-SLA-001 |
| Assembly date | 2026-06-22 |

### 7.4 Version control of the Assumption Schedule

| Scenario | Version action |
|---|---|
| New proposal | Version 1.0 |
| Pack version updated after first issue | Version 1.1 |
| Scope change requiring assumption addition/removal | Version 1.x |
| Major re-assembly (different pattern) | Version 2.0 |

The version field in the document header is updated with each reissue.

## 8. Deduplication Rules

### 8.1 ID-level duplicates

**Not possible by design.** All 13 assumption packs use unique ID prefixes. The Assembly Engine enforces Rule C (no duplication) at the ID level. No additional deduplication logic is required.

### 8.2 Suppression-level deduplication

Rules S1–S4 handle all known base-vs-overlay conflicts. When a suppression rule fires, the base assumption is removed and the overlay assumption appears. From the client's perspective, this is transparent — they see one coherent assumption, not two conflicting ones.

### 8.3 Semantic/topic-level overlap

Two packs may contain assumptions that address related (but distinct) aspects of the same topic. This is **intentional** and is NOT deduplicated. Example:
- ERP-SEC-001 (ERP security roles assumption) and OCI-SEC-001 (OCI IAM assumption) both address security but for distinct product layers. Both appear in the client schedule under Section F.

**Rule:** Do not deduplicate semantic overlaps. Let the assembled output stand. Both assumptions provide distinct commercial protection.

### 8.4 Manual deduplication (BU Lead only)

If a bid manager suspects a genuine duplication (two assumptions making the same claim about the same topic), they must escalate to BU Lead. The BU Lead may:
- Confirm both are intentional (different contexts)
- Remove one with a documented rationale in the Audit Report
- Request a pack amendment for future use

**The Assembly Engine does not perform semantic deduplication.** This is a human governance action.

## 9. Output Formats

### 9.1 WP17D target format — Proposal-Ready Markdown

The primary output of WP17D is **Proposal-Ready Markdown** (`.md` file). This format:

- Uses standard CommonMark markdown syntax
- Uses H2 headings for main sections (A through H)
- Uses H3 headings for sub-sections within Section C
- Uses numbered list syntax for assumption entries
- Is structured to convert cleanly to DOCX via standard markdown-to-Word tools
- Is version-controlled in the repository

### 9.2 DOCX integration target

When inserted into a proposal DOCX, the Assumption Schedule should map to these Word styles:

| Schedule element | Word style |
|---|---|
| Section heading (A, B, C...) | Heading 2 |
| Sub-section heading (C.1, C.2...) | Heading 3 |
| Assumption entry | List Number |
| Reference suffix `*(Ref: ID)*` | Character style: Italic or Small text |
| Preamble text | Normal |
| Context summary table | Table Grid |

The bid manager performs the DOCX insertion manually in WP17D. Automated DOCX generation is a future WP17E scope item.

### 9.3 Formats NOT in WP17D scope

| Format | Status |
|---|---|
| HTML | Not in WP17D scope — future enhancement |
| PDF | Generated from DOCX by bid manager; not engine output |
| Excel/CSV | Not in scope — Audit Report has structured data if needed |
| Direct DOCX generation | Not in WP17D scope — WP17E target |

### 9.4 File naming convention

| Document | File name |
|---|---|
| Client-facing schedule | `[TENDER_ID]_ASSUMPTION_SCHEDULE_V[X].md` |
| Internal audit report | `[TENDER_ID]_ASSEMBLY_AUDIT_REPORT.md` |

Examples:
- `PLENNEGY_ASSUMPTION_SCHEDULE_V1.md`
- `ARM_IT045_ASSUMPTION_SCHEDULE_V1.md`

## 10. What Does NOT Appear in the Client-Facing Schedule

The following are explicitly excluded from the client-facing schedule. They appear only in the Assembly Audit Report (internal):

| Element | Why excluded |
|---|---|
| Pack names and versions | Internal engineering detail; clients do not need to understand pack structure |
| Load order | Internal Assembly Engine concept |
| Suppression log | Internal rule processing; clients see only the final clean output |
| Suppression rule codes (S1, S2, S3, S4) | Internal rule identifiers |
| BOM codes | Internal product codes |
| Assembly pattern name | Internal classification (e.g., "EBS AMS Full Stack") |
| Pack assumption counts | Internal quality check metric |
| Rule A/B/C/D/E audit results | Internal rule compliance record |
| Gap assessment | Internal quality assessment against prior validation |
| BU Lead escalation items | Internal governance record |

## 11. Implementation Prerequisites for WP17D-1

Before WP17D-1 can generate compliant client-facing schedules, the ASSUMPTION_EXTRACTOR must be updated to:

1. Read full assumption text from each source pack file (not just IDs)
2. Apply the grouping mapping from `ASSUMPTION_GROUPING_RULES.md`
3. Apply sequential numbering across the document
4. Append the `*(Ref: ID)*` suffix to each entry
5. Suppress the presentation of any assumption marked as suppressed by the Rule Processor
6. Render replacement assumptions in their correct section position (not the suppressed assumption's position)
7. Omit all internal metadata from the client-facing output
8. Generate the document header block and preamble
9. Generate the engagement context summary from the tender profile
10. Omit empty sections (conditional sections with zero assumptions)
