## 2. Pre-Verification Defects Found and Resolved

During the initial WP18E-IMP-B verification pass, four defects were identified in the registry generated by Build 3. All four were fixed in `krpe.py` and confirmed resolved in Build 4.

### D-IMP-001 — ASM IDs Containing Em-Dash Annotations (BLOCKING → FIXED)

**Symptom:** 6 ASM entries in `KNOWLEDGE_ASSET_REGISTRY_ASSUMPTIONS.yaml` had `asset_id` values containing em-dash annotations, e.g., `EBS-SLA-005 — P1 Critical`. These are not valid IDs and would have caused KVE lookup failures.

**Root cause:** The `clean_id` stripping step in `extract_assumptions()` used `re.sub(r"\s+\[.*?\]", "", raw_id)` — removed bracket annotations `[...]` but not em-dash labels `— Label`.

**Fix (krpe.py):**
```python
clean_id = re.sub(r"\s+(?:\[.*?\]|[—–]\s*.+)$", "", raw_id).strip()
```

**Verification:** Zero ASM IDs containing em-dash character after Build 4. All 6 EBS SLA assumptions now registered with canonical IDs (`EBS-SLA-005` through `EBS-SLA-010`).

---

### D-IMP-002 — W5-METH-001 Empty `owner_business_unit` (WARNING → FIXED)

**Symptom:** CAP `W5-METH-001` (Cross-Platform Methodology) had empty `owner_business_unit` in the registry.

**Root cause:** The file uses `bu: "Cross-Platform (Oracle, Acumatica, OCI, OIC, BeBanking)"` rather than `business_unit:`. `_derive_bu()` only checked `business_unit`.

**Fix (krpe.py):**
```python
bu = _str(file_meta.get("business_unit", ""))
if not bu:
    bu = _str(file_meta.get("bu", ""))
    if bu:
        return _bu_from_raw(bu)
```

**Verification:** `W5-METH-001.owner_business_unit = "Corporate"` ✓

---

### D-IMP-003 — CAP `cap_ext.platform` Showing `Cross_BU` (WARNING → FIXED)

**Symptom:** 6 Corporate CAPs had `cap_ext.platform = "Cross_BU"` instead of `"Corporate"`.

**Root cause:** Platform derivation in `extract_cap()` read `business_unit` directly without applying `_BU_NORMALISE`.

**Fix (krpe.py):**
```python
raw_platform = _str(file_meta.get("business_unit", "") or file_meta.get("bu", "")) or owner_bu
platform = _BU_NORMALISE.get(raw_platform, raw_platform) if raw_platform in _BU_NORMALISE else owner_bu
```

**Verification:** All 7 Corporate CAPs show `cap_ext.platform = "Corporate"` ✓

---

### D-IMP-004 — Wave 1/2 CAPs with Empty `created` Field (WARNING → FIXED)

**Symptom:** 31 CAPs (all Wave 1/2) had empty `created` field. Wave 1/2 files use `extraction_date:` not `created:`.

**Fix (krpe.py):**
```python
"created": _str(file_meta.get("created", "") or file_meta.get("extraction_date", "")),
"created_by": _str(file_meta.get("created_by", "") or file_meta.get("extracted_by", "")),
"approved_by": _str(file_meta.get("approved_by", "") or file_meta.get("reviewed_by", "")),
```

**Verification:** `created` field populated for 48/49 CAPs after fix. Remaining 1 empty entry is `W2S1-004` whose source file is absent from the repository (OBS-001 below).

## 4. Observations (Non-Blocking)

### OBS-001 — W2S1-004 Source File Missing

**Asset:** `W2S1-004` (Oracle Managed Services)  
**Symptom:** `source_file: 07_Approved_Content/Approved/Oracle/W2S1-004-ORA-ManagedServices.md` does not exist on disk. Asset registered from MASTER_CAPABILITY_INDEX.md metadata only; field values are sparse.  
**Impact:** `created` and `approved_by` fields empty for this asset only. Asset is otherwise structurally valid and correctly registered.  
**Action required:** BU Lead to locate or recreate source file. Tracked as TD-IMP-001.

### OBS-002 — 19 Wave 3/4 ID Mismatches (Expected)

**Symptom:** KRPE emits `ID_MISMATCH` warnings for 19 Wave 3/4 CAPs where the MASTER_CAPABILITY_INDEX.md short ID (e.g., `W3S1-001`) differs from the file's `document_id` (e.g., `W3S1-001-ORA-HCMCore`).  
**Resolution:** By design — the file-level `document_id` is the governance-authoritative identifier. The registry correctly uses the descriptive document_id. The index should be updated to use descriptive IDs (TD-IMP-002).

### OBS-003 — 28 Wave 1/2 CAPs Missing `wave` Field (Expected)

**Cause:** Wave 1/2 files use an old frontmatter format that does not include a `wave:` field. Not present to extract.  
**Impact:** None — wave can be inferred from asset_id prefix (W1/W2).

### OBS-004 — 34 Wave 1/2 CAPs Missing `deliverable` Field (Expected)

**Cause:** Same old frontmatter format as OBS-003. No `deliverable:` field in Wave 1/2 files.  
**Impact:** None — deliverable information absent from source; not a registry defect.

### OBS-005 — 16 CAPs Missing `approval_date` (Informational)

**Cause:** 15 Wave 1/2 files lack both `approval_date:` and `review_date:` fields in frontmatter. W2S1-004 has no source file.  
**Impact:** Informational only. All 49 CAPs are status=APPROVED and approved_for_reuse=true; the date field is supplementary metadata.

### OBS-006 — 3 Packs Missing Declared `assumption_count` in Frontmatter (Low)

**Packs:** HCM-LEARNING-ASSUMPTIONS-V1, HCM-TALENT-ASSUMPTIONS-V1, EBS-DRM-ASSUMPTIONS-V1  
**Impact:** None — registry uses actual extracted count, which is authoritative. Tracked as TD-IMP-003/004.

### OBS-007 — All 49 CAPs Are Relationship Orphans (Expected)

**Cause:** Phase A populates only REL-001 (ASP→ASM containment). CAP→SEC, SEC→CAP, and ASP→CAP relationships are Phase B scope. No CAP appears as a relationship source or target in Phase A.  
**Impact:** None in Phase A. KVE rules that traverse CAP→ASP links are deferred to Phase B.
